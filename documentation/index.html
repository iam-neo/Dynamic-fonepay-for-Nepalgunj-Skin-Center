1. The Problem Space

Standard printed QR codes found at Nepali merchant locations (like Nepalgunj Skin Center) contain static Fonepay/EMVCo payloads. When scanned by a banking application (e.g., eSewa, Global Smart, Khalti), the user must manually enter the correct payment amount.

Our objective was to build a secure, dynamic QR generator that dynamically modifies the payload to embed an exact decimal amount. This ensures precision, eliminates manual entry errors by the customer, and accelerates checkout times.

2. The Tech Stack

We completely migrated the legacy single-file HTML version to a modern Single Page Application (SPA) architecture:

  • Framework: React 19 for component-based logic and state management.
  • Build Tool: Vite 8, chosen for lightning-fast Hot Module Replacement (HMR) and optimized build times.
  • Styling: Tailwind CSS v4, allowing us to build a rich, complex glassmorphism UI entirely with utility classes.
  • QR Generation: qrcode.react to handle high-error-correction canvas drawing, allowing us to embed the company logo perfectly in the center.
  • Typography: Inter (Google Fonts) for clean, modern, highly legible text across all screen sizes.
  • Deployment: Vercel edge network with immutable asset caching and SPA rewrites.

3. Decoding the EMVCo Protocol

Fonepay utilizes the EMVCo QR code specification. The payload is structured as a series of Tag-Length-Value (TLV) nodes. Our generator constructs the following tags:

EMVCo Tag Map 

  Tag   Field                        Value
  ────  ───────────────────────────   ──────────────────────────
  00    Payload Format Indicator      01 (version 1)
  01    Point of Initiation           12 (Dynamic — enables auto-fill)
  26    FonePay Merchant Data         fonepay.com + Merchant Code + Terminal
  37    Merchant Identifier           9105240052044
  52    Merchant Category Code        8012
  53    Currency Code                 524 (NPR)
  54    Transaction Amount            ← User input (e.g., 500.00)
  58    Country Code                  NP
  59    Merchant Name                 NEPALGUNJ SKIN CENTER
  60    Merchant City                 Banke
  62    Additional Data               Terminal ID + TxnID + Remarks
  63    CRC Checksum                  CRC16-CCITT (4-char HEX)
generateQRString.js — Tag Construction 

export function generateQRString(amount, remarks = 'Payment') {
    let qr = '000201';           // Payload Format Indicator
    qr += '010212';              // Dynamic (not 010211 Static)
    qr += '37139105240052044';   // Merchant Identifier

    // Tag 26: FonePay merchant data block
    qr += '26390011fonepay.com011021090206640206497140';

    qr += '52048012';            // Category Code
    qr += '5303524';             // Currency: NPR

    // Tag 54: Transaction Amount (enables auto-fill in banking apps)
    const amtStr = amount.toFixed(2);
    qr += '54' + amtStr.length.toString().padStart(2, '0') + amtStr;

    qr += '5802NP';              // Country
    qr += '5921NEPALGUNJ SKIN CENTER'; // Merchant Name
    qr += '6005Banke';           // City

    // Tag 62: Additional Data (Terminal + TxnID + Remarks)
    const txnId = 'TXN' + Date.now().toString().slice(-6);
    ...
    // Tag 63: CRC16-CCITT checksum
    qr += '6304';
    const crc = crc16(qr);
    qr += crc;

    return { qrString: qr, amtStr, crc, txnId };
}
Critical: Tag 01 must be 12 (Dynamic), not 11 (Static). Without this, banking apps will ignore the amount tag entirely and require manual entry.

4. Checksum Geometry (CRC16-CCITT)

The most complex part of a dynamic EMVCo QR is the Checksum (Tag 63). If you modify a single byte (like the amount or remarks), the banking app will reject the entire QR code unless the 4-character HEX checksum is perfectly recalculated.

We wrote a robust, zero-dependency CRC16-CCITT algorithm in JavaScript to handle this dynamically on the client side:

crc16.js — Complete Implementation 

export function crc16(str) {
    let crc = 0xffff;
    for (let i = 0; i < str.length; i++) {
        crc ^= str.charCodeAt(i) << 8;
        for (let j = 0; j < 8; j++) {
            crc = crc & 0x8000
                ? (crc << 1) ^ 0x1021
                : crc << 1;
        }
    }
    return (crc & 0xffff)
        .toString(16)
        .toUpperCase()
        .padStart(4, '0');
}
Polynomial 0x1021: The algorithm seeds with 0xFFFF, XORs each byte shifted left by 8, and iterates 8 bits per character through the standard CCITT polynomial — producing a 4-digit HEX string appended as Tag 63.

8. Smart Amount Formatting

One common real-world problem in payment terminals is amount entry mistakes. When typing large numbers like 100000, it's difficult to visually confirm whether you typed one lakh or ten thousand.

We solved this by implementing live Indian/Nepali number formatting directly inside the amount input field. As the operator types, the digits are automatically formatted with commas following the South Asian numbering system:

Live Formatting Examples 

  Typed         Displayed
  ──────────    ─────────────
  1000      →   1,000
  10000     →   10,000
  100000    →   1,00,000
  1000000   →   10,00,000
  10000000  →   1,00,00,000
QRForm.jsx — formatIndianNumber() 

function formatIndianNumber(num) {
    if (!num) return '';
    const parts = num.split('.');
    let intPart = parts[0];
    const decPart = parts.length > 1 ? '.' + parts[1] : '';

    intPart = intPart.replace(/^0+(?=\d)/, '');  // strip leading zeros

    if (intPart.length <= 3) return intPart + decPart;

    // Last 3 digits get one comma, then every 2 digits
    const last3 = intPart.slice(-3);
    const rest  = intPart.slice(0, -3);
    const formatted = rest.replace(/\B(?=(\d{2})+(?!\d))/g, ',')
                         + ',' + last3;
    return formatted + decPart;
}
Dual-State Architecture: The input uses type="text" with inputMode="decimal" (mobile numeric keyboard). Two separate React state values are maintained — displayAmount (formatted with commas for the UI) and amount (raw digits for QR generation). This ensures visual clarity without corrupting the numeric payload.

9. Continuous Deployment

We mapped the project specifically for Vercel edge deployments. The site instantly goes live whenever code is pushed to the GitHub master branch.

vercel.json — SPA Rewrites & Asset Caching 

{
    "rewrites": [
        { "source": "/(.*)", "destination": "/index.html" }
    ],
    "headers": [
        {
            "source": "/assets/(.*)",
            "headers": [
                {
                    "key": "Cache-Control",
                    "value": "public, max-age=31536000, immutable"
                }
            ]
        }
    ]
}

The rewrite rule ensures all client-side routes resolve to index.html (SPA fallback). The cache header sets a 1-year immutable cache on Vite's hashed assets in /assets/ for optimal CDN performance.

.npmrc: We configured legacy-peer-deps=true to bypass strict peer dependency checks between Vite 8 and Tailwind CSS v4 ecosystem tools, ensuring reliable remote builds on Vercel's CI pipeline.

10. How the Code Works

The application is built on a modular React architecture that strictly separates presentation from complex EMVCo payment logic. Here is the complete lifecycle of a QR code generation:

Data Flow & Component Architecture

Component Tree 

App.jsx (Root — State: qrData, showModal)
 ├── Header.jsx
 │    └── "fonepay" branding + NRB Nepali authorization text
 ├── MerchantCard.jsx
 │    └── Nepalgunj Skin Center | Code: 2109020664 | Terminal: 497140
 ├── QRForm.jsx
 │    ├── Amount Input (live Indian/Nepali comma formatting)
 │    ├── Remarks Input (optional)
 │    ├── Quick-Fill Presets (8 service buttons)
 │    └── Generate Button → triggers onGenerate(amount, remarks)
 └── QR Modal Popup (glassmorphism overlay)
      └── QRDisplay.jsx
           ├── <QRCodeCanvas /> with logo overlay (Level-H error correction)
           └── "Amount auto-populates in banking app" confirmation
  • 1. State Management (App.jsx): The root component holds two state variables — qrData (the generated payload) and showModal (popup visibility). When the user generates a QR, both are updated simultaneously.
  • 2. User Input (QRForm.jsx): This component manages three pieces of state — raw amount, display amount (with Indian commas), and remarks. It sanitizes input to allow only digits and a single decimal point. Quick-fill presets auto-populate both fields.
  • 3. Payload Generation (utils/generateQRString.js): The brain of the app. It takes the float amount and remarks, constructs all 12+ TLV tags in sequence, generates a unique transaction ID (TXN + last 6 digits of timestamp), and builds the complete EMVCo string.
  • 4. Checksum Calculation (utils/crc16.js): Before the final string is returned, it passes through the CRC16-CCITT algorithm to generate a mathematically sound 4-digit Hex verification signature appended as Tag 63.
  • 5. Modal Display (App.jsx + QRDisplay.jsx): The generated qrData triggers a glassmorphism overlay modal. Inside, QRDisplay renders a Level-H <QRCodeCanvas> (220px, with logo excavation) and a green confirmation badge.
