1. The Problem Space

Standard printed QR codes found at Nepali merchant locations (like Nepalgunj Skin Center) contain static Fonepay/EMVCo payloads. When scanned by a banking application (e.g., eSewa, Global Smart, Khalti), the user must manually enter the correct payment amount.

Our objective was to build a secure, dynamic QR generator that dynamically modifies the payload to embed an exact decimal amount. This ensures precision, eliminates manual entry errors by the customer, and accelerates checkout times.

2. The Tech Stack

We completely migrated the legacy single-file HTML version to a modern Single Page Application (SPA) architecture:

  • Framework: React 18 for component-based logic and state management.
  • Build Tool: Vite 8, chosen for lightning-fast Hot Module Replacement (HMR) and optimized build times.
  • Styling: Tailwind CSS v4, allowing us to build a rich, complex glassmorphism UI entirely with utility classes.
  • QR Generation: qrcode.react to handle high-error-correction canvas drawing, allowing us to embed the company logo perfectly in the center.

3. Decoding the EMVCo Protocol

Fonepay utilizes the EMVCo QR code specification. The payload is structured as a series of Tag-Length-Value (TLV) nodes.

generateQRString.js (Snippet) 

// Tag 54 represents the Transaction Amount
const formatTag = (id, value) => {
    const len = value.length.toString().padStart(2, '0');
    return `${id}${len}${value}`;
};

// Generating the dynamic part of the payload
let amountStr = formatTag("54", amount.toFixed(2));

To inject the amount dynamically, we dissect the static string, remove any existing Tag 54 nodes, construct a new Tag 54 using the user's target amount (e.g., 540510.00), and inject it right before Tag 63 (the Checksum tag).

4. Checksum Geometry (CRC16-CCITT)

The most complex part of a dynamic EMVCo QR is the Checksum (Tag 63). If you modify a single byte (like the amount or remarks), the banking app will reject the entire QR code unless the 4-character HEX checksum is perfectly recalculated.

We wrote a robust, zero-dependency CRC16-CCITT algorithm in JavaScript to handle this dynamically on the client side:

Polynomial 0x1021: The algorithm seeds with 0xffff and XORs each byte, shifting 8 bits at a time through the standard CCITT polynomial.

6. Continuous Deployment

We mapped the project specifically for Vercel edge deployments. By using Vite and defining client-side routing rewrites inside vercel.json, the site instantly goes live whenever code is pushed to the GitHub master branch.

We also implemented an .npmrc configuration to bypass strict peer dependency checks between Vite 8 and Tailwind ecosystem tools, ensuring robust remote builds.

7. How the Code Works

The application is built on a modular React architecture that strictly separates presentation from complex EMVCo payment logic. Here is the lifecycle of a QR code generation within our app:

Data Flow & Component Architecture

Component Tree 

App.jsx (Main Container & State Holder)
 ├── Header.jsx (Static text & Logos)
 ├── MerchantCard.jsx (Merchant Name & Terminal Details)
 ├── QRForm.jsx (User Inputs)
 │    └── Triggers onGenerate(amount, remarks)
 └── QRDisplay.jsx (QR Canvas & Data Viewer)
      ├── Renders 
      └── Displays raw string, Amount Tag, Checksum
  • 1. State Management (App.jsx): The root component holds a single state variable: qrData. When a user interacts with the generator, App.jsx passes the updated payload down to the display component.
  • 2. User Input (QRForm.jsx): This component handles controlled inputs for the payment amount and optional remarks. Upon submission, it sanitizes the number and triggers an onGenerate callback function passed from the parent App.jsx.
  • 3. Payload Generation (utils/generateQRString.js): This is the brain of the app. It takes the float amount and remarks, constructs missing TLV tags (Tag 54 for Amount, Tag 08 for Remarks), and splices them into the massive static EMVCo string.
  • 4. Checksum Calculation (utils/crc16.js): Before the final string is returned, the app passes the newly constructed string (minus the final checksum) through our custom CRC16-CCITT algorithm to generate a mathematically sound 4-digit Hex verification signature.
  • 5. Rendering (QRDisplay.jsx): Finally, the new qrData object is passed into QRDisplay. We use qrcode.react to draw a Level-H (High Error Correction) canvas, which is dense enough to allow us to safely overlay the `logo.png` image directly over the center data modules without breaking scannability.
